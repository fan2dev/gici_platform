## Project Overview

GICI Platform is a multitenant SaaS for childcare centers. One Serverpod backend + one Flutter app (mobile & web) + an auto-generated client library.

## Monorepo Layout

- `gici_backend_server/` — Serverpod backend (Dart). Endpoints, services, models (.spy.yaml), migrations.
- `gici_backend_client/` — Auto-generated Dart client (do not edit manually; regenerated by `serverpod generate`).
- `gici_app/` — Flutter client (mobile + web). Uses BLoC/Cubit, GoRouter, GetIt.
- `docs/` — Architecture, auth, multitenancy, domain model, and feature documentation.

## Development Commands

### Backend setup and run

```bash
cd gici_backend_server
docker compose up --build --detach   # PostgreSQL 16 (pgvector) on :8090, Redis on :8091
dart pub get
serverpod generate                   # Regenerate protocol + client from .spy.yaml models
serverpod create-migration           # After model changes
dart bin/main.dart --apply-migrations  # Start server + apply migrations (API :8080, Insights :8081)
```

### Flutter app

```bash
cd gici_app
flutter pub get
flutter run -d chrome    # or: -d ios, -d android
```

### Linting

```bash
dart analyze             # Backend (from gici_backend_server/)
flutter analyze          # App (from gici_app/)
```

### Testing

```bash
# Backend tests (docker compose must be running for test postgres/redis on :9090/:9091)
cd gici_backend_server
dart test
dart test test/path/to/specific_test.dart   # Single test file

# Flutter tests
cd gici_app
flutter test
flutter test test/path/to/specific_test.dart
```

## Architecture Essentials

### Multitenancy

All business tables carry `organizationId`. Every query must filter by it. Endpoints receive both `organizationId` and `actorId` as parameters.

### Authorization pattern

Actor-based, not client role-string based. `AccessControlService.requireActor(...)` validates identity, active status, role, and org membership. Four roles: `platform_super_admin` (cross-tenant), `organization_admin`, `staff`, `guardian` (restricted to linked children and participant-only conversations).

### Endpoint → Service layer

Endpoints are thin; business logic lives in service classes under `gici_backend_server/lib/src/services/`. Endpoints call services, services query the database.

### Code generation flow

Model definitions live in `.spy.yaml` files under `gici_backend_server/lib/src/models/`. Running `serverpod generate` regenerates:
- `gici_backend_server/lib/src/generated/` (server protocol)
- `gici_backend_client/lib/src/protocol/` (client protocol)

Never edit generated files directly.

### Flutter state management

- **BLoC/Cubit** for state management in `gici_app/lib/features/`
- **GoRouter** with role-based route guards (guardians blocked from time-tracking, classrooms)
- **GetIt** for DI (`gici_app/lib/core/di/injection.dart`)
- Session persisted via `AuthSessionStorage` (stores `appUserId`); `AuthCubit.bootstrap()` calls `auth.me` at startup

### Auth flow

Login: `auth.signInWithEmailPassword(email, password)` → returns `AuthSession` DTO. Passwords hashed server-side. Endpoints never trust client-provided role claims. Signed session tokens are planned but not yet implemented; current pattern passes `actorId` in request params.

### Key conventions

- Pagination: `page` + `pageSize` parameters on list endpoints
- Audit: operations logged to `ActivityLog` table (append-only)
- Some enum fields stored as `String` for model generation stability
- Soft-delete via `deletedAt` where operationally needed
- DTO objects for complex views (e.g., `ChildProfileOverview`, `ChildDailyHabits`)
