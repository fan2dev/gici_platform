## Project Overview

GICI Platform is a multitenant SaaS for childcare centers. One Serverpod backend + one Flutter app (mobile & web) + an auto-generated client library.

## Monorepo Layout

- `gici_backend_server/` — Serverpod backend (Dart). Endpoints, services, models (.spy.yaml), migrations.
- `gici_backend_client/` — Auto-generated Dart client. **NEVER edit manually**; regenerated by `serverpod generate`.
- `gici_app/` — Flutter client (mobile + web). Uses BLoC/Cubit, GoRouter, GetIt.
- `docs/` — Architecture, auth, multitenancy, domain model, and feature documentation.

## Development Commands

### Backend setup and run

```bash
cd gici_backend_server
docker compose up --build --detach   # PostgreSQL 16 (pgvector) on :8090, Redis on :8091
dart pub get
serverpod generate                   # Regenerate protocol + client from .spy.yaml models
serverpod create-migration           # After model changes
dart bin/main.dart --apply-migrations  # Start server + apply migrations (API :8080, Insights :8081)
```

### Flutter app (flavors)

```bash
cd gici_app
flutter pub get
flutter run -t lib/main_gici.dart -d chrome       # GICI white-label (web)
flutter run -t lib/main_borreguet.dart -d chrome   # Borreguet flavor (web)
flutter run -t lib/main_gici.dart                   # GICI (mobile)
```

### Linting

```bash
dart analyze             # Backend (from gici_backend_server/)
flutter analyze          # App (from gici_app/)
```

### Seed demo data

Call endpoint `bootstrap.seedDemoData` with `bootstrapKey: local-dev-bootstrap`. Seeds "El Borreguet" as the first tenant.

## Critical Rules

### Model changes workflow
All model changes MUST go through `.spy.yaml` files → `serverpod generate`. **NEVER edit files in `generated/` directories** — they will be overwritten.

### UUID Primary Keys
All models use `UuidValue` as primary key. In `.spy.yaml`:
```yaml
fields:
  id: UuidValue?, defaultPersist=random
```
Use `defaultPersist`, NOT `default`. Foreign keys also use `UuidValue` type.

### Serverpod Auth uses int IDs internally
The `serverpodUserId` field linking `AppUser` to Serverpod's `UserInfo` is `int?` because Serverpod Auth's internal tables use integer IDs.

## Architecture Essentials

### Authentication (Serverpod Auth)

- Server initializes Serverpod Auth in `server.dart` via `AuthConfig.set(...)`
- Sign in: validates credentials against `AppUser`, creates Serverpod `UserInfo` + auth key via `UserAuthentication.signInUser()`
- Session: endpoints use `getAuthenticatedUser(session)` helper which calls `session.authenticated` to get the Serverpod user ID, then looks up `AppUser` by `serverpodUserId`
- Flutter client: `FlutterAuthenticationKeyManager` handles automatic token persistence
- Password recovery: Serverpod Auth's built-in email reset flow (`Emails.initiatePasswordReset`, `Emails.resetPassword`)
- Passwords hashed with bcrypt

### Authorization pattern

`AccessControlService.requireRole(actor, allowedRoles: [...])` validates role. `requireGuardianChildAccess()` checks guardian-child link. Four roles: `platform_super_admin` (cross-tenant), `organization_admin`, `staff`, `guardian`.

### Multitenancy

All business tables carry `organizationId: UuidValue`. The actor's `organizationId` is read from the authenticated `AppUser` — never passed as a parameter.

### Endpoint → Service layer

Endpoints are thin; business logic lives in service classes under `gici_backend_server/lib/src/services/`.

### Flavors + ConfigurationService

- `AppConfig` in `gici_app/lib/core/config/app_config.dart` holds per-flavor config
- Entry points: `main_gici.dart`, `main_borreguet.dart` — each calls `bootstrap(config)`
- Config includes: appName, bundleId, apiUrl, seedColor

### Flutter state management

- **BLoC/Cubit** per feature in `gici_app/lib/features/*/cubit/`
- **GoRouter** with ShellRoute for adaptive shell (sidebar on web, bottom nav on mobile)
- **GetIt** for DI (`gici_app/lib/core/di/injection.dart`)
- Adaptive UI shell in `gici_app/lib/app/shell/` detects web vs mobile via `LayoutBuilder`
- Shared widgets in `gici_app/lib/app/widgets/` (AdaptivePage, EmptyState, ErrorState, LoadingState)

### Key conventions

- All IDs are `UuidValue` (UUID v4, generated by PostgreSQL `gen_random_uuid()`)
- Pagination: `page` + `pageSize` parameters on list endpoints
- Audit: operations logged to `ActivityLog` table (append-only)
- Soft-delete via `deletedAt` where operationally needed
- DTO objects for complex views (e.g., `ChildProfileOverview`, `DashboardSummary`)
